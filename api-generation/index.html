    HollowAPIGenerator File Generation Guide¶

    Overview¶

    HollowAPIGenerator.generateFiles generates a complete Java API layer for consuming Hollow datasets. This document explains what files are generated and the purpose of each file type.

    Top-Level API Files¶

    These files are generated once per API, regardless of the number of schema types.

    1. {ApiClassName}.java (e.g., TestSchemaAPI.java)¶

    Purpose: Main entry point API class that extends HollowAPI

    • Initializes and manages all type-specific APIs
    • Provides factory methods to instantiate Hollow objects by ordinal (e.g., getObject1(int ordinal))
    • Manages object providers (cached vs factory-based)
    • Provides getAll{TypeName}() methods to iterate over all records of a type
    • Handles sampling and caching lifecycle

    Example methods: 

    public Object1 getObject1(int ordinal)
public Collection<Object1> getAllObject1()
public void detachCaches()

    2. {ApiClassName}Factory.java (e.g., TestSchemaAPIFactory.java)¶

    Purpose: Factory for creating API instances

    • Creates the main API object from a HollowDataAccess
    • Simplifies API instantiation

    3. {ApiClassName}HashIndex.java (e.g., TestSchemaAPIHashIndex.java)¶

    Purpose: Generic hash index builder for ad-hoc queries

    • Allows building custom hash indexes on any type
    • Provides find{TypeName}Matches() methods for each schema type
    • Useful for creating indexes on fields not covered by primary keys

    Example methods: 

    public Iterable<Object1> findObject1Matches(Object... keys)
public Iterable<TopLevelObject> findTopLevelObjectMatches(Object... keys)

    Per-Type Files¶

    These files are generated for each schema type in your data model (e.g., Object1, String, ListOfObject1, etc.).

    4. {TypeName}.java (e.g., Object1.java)¶

    Purpose: The main business object representing a record in the dataset

    • Extends HollowObject
    • Provides strongly-typed getter methods for all fields (e.g., getVal())
    • Methods delegate to the underlying delegate for data access
    • Provides access to the API and TypeAPI

    Example: 

    public class Object1 extends HollowObject {
    public HString getVal() {
        int refOrdinal = delegate().getValOrdinal(ordinal);
        if(refOrdinal == -1)
            return null;
        return api().getHString(refOrdinal);
    }

    public TestSchemaAPI api() {
        return typeApi().getAPI();
    }
}

    5. {TypeName}TypeAPI.java (e.g., Object1TypeAPI.java)¶

    Purpose: Type-specific API for low-level data access

    • Extends HollowObjectTypeAPI (or HollowListTypeAPI, HollowSetTypeAPI, HollowMapTypeAPI)
    • Provides ordinal-based field access (e.g., getValOrdinal(int ordinal))
    • Maps field names to field indices
    • Provides references to related type APIs

    Example methods: 

    public int getValOrdinal(int ordinal)
public StringTypeAPI getValTypeAPI()

    6. {TypeName}HollowFactory.java (e.g., Object1HollowFactory.java)¶

    Purpose: Factory for creating instances of the Hollow object

    • Creates new hollow objects (lookup-based implementation)
    • Creates cached hollow objects (snapshot-based implementation)
    • Used by the API's object provider system

    Example methods: 

    public T newHollowObject(HollowTypeDataAccess dataAccess, HollowTypeAPI typeAPI, int ordinal)
public T newCachedHollowObject(HollowTypeDataAccess dataAccess, HollowTypeAPI typeAPI, int ordinal)

    Object Schema-Specific Files¶

    These files are only generated for OBJECT type schemas (not for LIST, SET, or MAP types).

    7. {TypeName}Delegate.java (e.g., Object1Delegate.java)¶

    Purpose: Interface for accessing object data

    • Defines methods to get ordinals of referenced fields
    • Allows different implementations (lookup vs cached)
    • Abstracts the data access strategy

    Example: 

    public interface Object1Delegate extends HollowObjectDelegate {
    public int getValOrdinal(int ordinal);
    public Object1TypeAPI getTypeAPI();
}

    8. {TypeName}DelegateCachedImpl.java (e.g., Object1DelegateCachedImpl.java)¶

    Purpose: Caches field values at object creation time

    • Reads all field values eagerly when object is created
    • Faster repeated access but uses more memory
    • Useful for objects accessed multiple times
    • Immutable snapshot of the object's state

    Use case: When you need to access the same object's fields many times

    9. {TypeName}DelegateLookupImpl.java (e.g., Object1DelegateLookupImpl.java)¶

    Purpose: Looks up field values on-demand

    • Reads field values lazily from the dataset
    • Lower memory footprint
    • Good for one-time access patterns
    • Always reflects current dataset state

    Use case: When you access objects once or infrequently

    10. {TypeName}DataAccessor.java (e.g., Object1DataAccessor.java)¶

    Purpose: High-level accessor for iterating and querying objects

    • Extends AbstractHollowDataAccessor
    • Provides convenient access to records by ordinal
    • Used for bulk operations and traversals
    • Integrates with HollowConsumer

    Example usage: 

    Object1DataAccessor accessor = new Object1DataAccessor(consumer);
Object1 obj = accessor.getRecord(ordinal);

    11. {TypeName}PrimaryKeyIndex.java (e.g., Object1PrimaryKeyIndex.java)¶

    Purpose: Unique key index for lookups

    • Generated when:
    • reservePrimaryKeyIndexForTypeWithPrimaryKey is false (legacy mode - always generated), OR
    • reservePrimaryKeyIndexForTypeWithPrimaryKey is true AND the schema has a primary key defined
    • Provides findMatch(Object... keys) method for unique key lookups
    • Automatically listens to data refreshes to stay up-to-date
    • Marked as @Deprecated in favor of newer UniqueKeyIndex API

    Example usage: 

    Object1PrimaryKeyIndex index = new Object1PrimaryKeyIndex(consumer, "val");
Object1 obj = index.findMatch("someValue");

    12. {TypeName}UniqueKeyIndex.java¶

    Purpose: Modern unique key index (when primary key exists)

    • Only generated when reservePrimaryKeyIndexForTypeWithPrimaryKey is true AND schema has a primary key
    • Provides similar functionality to PrimaryKeyIndex but with newer API patterns
    • Recommended for new code

    File Count by Schema Type¶

    Schema Type Number of Generated Files Files
    OBJECT (no primary key) 9 files TypeName.java, TypeNameTypeAPI.java, TypeNameHollowFactory.java, TypeNameDelegate.java, TypeNameDelegateCachedImpl.java, TypeNameDelegateLookupImpl.java, TypeNameDataAccessor.java, TypeNamePrimaryKeyIndex.java (legacy mode only)
    OBJECT (with primary key) 10-11 files Above + TypeNamePrimaryKeyIndex.java and TypeNameUniqueKeyIndex.java (depending on config)
    LIST 3 files TypeName.java, TypeNameTypeAPI.java, TypeNameHollowFactory.java
    SET 3 files TypeName.java, TypeNameTypeAPI.java, TypeNameHollowFactory.java
    MAP 3 files TypeName.java, TypeNameTypeAPI.java, TypeNameHollowFactory.java

    Summary of File Purposes by Category¶

    File Category Purpose
    Main API Entry point for consuming data, manages all types
    Hollow Objects Business objects with type-safe getters
    Type APIs Low-level ordinal-based field access
    Factories Create object instances (cached or lookup-based)
    Delegates Abstract data access (cached vs lookup strategies)
    Data Accessors High-level iterators and bulk operations
    Indexes Efficient lookups by primary/unique keys or hash-based queries

    Configuration Options¶

    The HollowAPIGenerator can be configured with various options that affect which files are generated:

    • setClassPostfix(String): Changes the postfix for generated class names (default: "Hollow")
    • setGetterPrefix(String): Changes the prefix for getter methods (default: "_")
    • setUsePackageGrouping(boolean): Organizes files into sub-packages
    • reservePrimaryKeyIndexForTypeWithPrimaryKey(boolean): Only generate PrimaryKeyIndex for types with primary keys defined
    • setUseHollowPrimitiveTypes(boolean): Use built-in primitive types instead of generating them
    • setUseBooleanFieldErgonomics(boolean): Use improved boolean field naming conventions
    • setRestrictApiToFieldType(boolean): Restrict generated API methods to exact field types

    Typical Usage Pattern¶

    1. Generate the API: 

      HollowAPIGenerator generator = new HollowAPIGenerator.Builder()
    .withAPIClassname("MyAPI")
    .withPackageName("com.example.api")
    .withDataModel(writeStateEngine)
    .withDestination("/path/to/output")
    .build();
generator.generateSourceFiles();

    2. Consume data: 

      // Initialize
TestSchemaAPI api = new TestSchemaAPI(dataAccess);

// Access objects
TopLevelObject obj = api.getTopLevelObject(ordinal);
Object1 member = obj.getValidMemberField();
HString val = member.getVal();

// Use indexes
Object1PrimaryKeyIndex index = new Object1PrimaryKeyIndex(consumer, "val");
Object1 found = index.findMatch("searchKey");

    Benefits¶

    This generated code provides:

    • Type safety: Compile-time checking of field access
    • Performance: Efficient ordinal-based access with optional caching
    • Flexibility: Choice between cached and lookup delegates
    • Convenience: High-level APIs for common operations
    • Efficiency: Optimized indexes for fast lookups
    • Maintainability: Generated code stays in sync with schema changes
